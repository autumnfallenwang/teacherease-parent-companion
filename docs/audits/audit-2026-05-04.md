## Watch list

### `src/components/settings-email-section.tsx` (1/5 signals)

**Signals firing (1/5):** defensive-justification (weak)

- **Bug-ratio inversion:** no — 4 feat/refactor vs 1 fix across 5 commits.
- **Edge-case avalanche:** no — fix history is a single SMTP-recipient regex tweak, not a class-of-problem cascade.
- **Foundation-becomes-ceiling:** no — the file is the *only* in-scope component with the hand-rolled `validate()` + `errors: Partial<Record<string, string>>` pattern (`settings-email-section.tsx:229,250`). That's the standard early shape of a form before RHF; it isn't blocking any feature today.
- **Knowledge isolation:** no — structure is clear: top-level `SettingsEmailSection` + `SmtpSummaryCard` + `EmailForm` sub-components, each ≤120 lines.
- **Defensive justification:** weak yes — the file carries SMTP config + recipient list parsing (`splitTo`/`joinTo`) + test-send + per-field error tracking. None is bad in isolation, but the validation surface is the only place in the app where a hand-rolled `errors` object is being maintained, and that pattern doesn't compose well if a second multi-field form (e.g. per-child SMTP overrides, or an OAuth provider config) ever lands.

**Why not swap now:**
- 1 signal, not 2. The skill rule is "default to keep unless 2+ fire."
- The form works. There is no current feature blocked by its shape.
- `react-hook-form + zod` (~30 KB) is real cost; ~480 lines of working code is real value to delete.

**Trigger to revisit at next audit:**
- A second multi-field form lands and adopts the same pattern → signal #2 (edge-case avalanche) starts firing.
- A bug appears in validation drift between save path and test-send path → signal #1 starts firing.
- Q-decision adds per-child SMTP or multi-account → signal #3 (foundation-as-ceiling) starts firing.

If any of these happen, this file flips to a swap recommendation against `react-hook-form + zod` per `references/alternatives.md`.
